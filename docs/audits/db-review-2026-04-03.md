## Warnings (non-blocking for MVP)

### ⚠️ 1. `correct_key` exposed via anon key
`questions` has no RLS. Anyone with the `publishable_key` can query every correct answer via REST. For an exam-prep MVP this is common and acceptable (answers are in any book), but worth noting if you want to prevent bots from scraping the full bank. Future fix: if you want to filter visibility, use RLS to expose only valid rows (e.g. `is_active = true`), but not as a "row-limiting" mechanism. To meaningfully reduce scraping, serve questions via an RPC/Edge Function that returns a random subset, and/or expose a public view that omits `correct_key`.

### ⚠️ 2. `readiness_scores` not created in the signup trigger
The `on_auth_user_created` trigger only inserts into `profiles`. The `readiness_scores` row has to be created elsewhere (probably on first session or via an upsert from the app). Consider adding it to the trigger to guarantee consistency.

### ⚠️ 3. Question bank insufficient for launch
The seed has 5 questions. `random_10` mode needs at least 10 per domain, and `full_exam` needs 90. This is a content task, not a code task — but it blocks launch.

### ℹ️ 4. `subscription_status` and `subscription_tier` are `text` with no constraint
Consider adding a `check` constraint to guarantee valid values. Under load, a bug in the Stripe webhook could write an invalid state.

## Executive summary

The schema is well designed for the MVP scope. The separation between public content (`exams`, `domains`, `questions`) and user data (`profiles`, `user_responses`, `quiz_sessions`, `readiness_scores`) is correct. RLS is enabled where it must be. The SM-2 model is correctly embedded in `user_responses`. Indexes cover the critical access patterns.

**Before launch:**
1. Expand question bank to 100+
2. Add `readiness_scores` insert to the `on_auth_user_created` trigger
3. Decide whether to add RLS to `questions` (optional for MVP)
